Product Requirements Document
FormForge — Dynamic Schema-Driven Form Platform

FormForge eliminates the repetitive work of building HTML forms, validation logic, and state management. You define a JSON schema — FormForge generates a fully functional, validated, and beautiful form automatically.

Design Document
Architecture, core flows and system design

FormForge is built around one principle: the schema is the single source of truth for a form's structure, behavior, validation, and output. Every subsystem derives its behavior from that schema at runtime.

v1.0 In review Engineering · internal
System modules
@formforge/parser
Schema Parser
Validates, normalizes, and indexes the input schema. Catches schema-level errors before any rendering occurs. The first stage in the pipeline — nothing proceeds with an invalid schema.
@formforge/state
State Manager
Centralizes all field values and dirty/pristine tracking. The single source of truth at runtime. All mutations go through a single dispatcher — no hidden state.
@formforge/engine
Execution Engine
Evaluates conditional logic and triggers validation on every state change. Coordinates all other modules. The brain of the system.
@formforge/react
Renderer (React)
Consumes engine output and renders the appropriate field components. Framework-specific adapter — the core engine is framework-agnostic.
@formforge/serializer
Payload Serializer
Transforms internal state into a clean submission-ready object. Hidden and conditionally-invisible fields are stripped from the output automatically.

User interaction triggers onChange(fieldId, value) → the engine updates state, re-evaluates all conditions referencing the changed field, runs validation, and the renderer reflects the new state. Only affected fields re-render.

Schema shape
// formforge.schema.js const schema = { fields: [ { id: "email", type: "email", label: "Email address", required: true, validation: [ { rule: "format", value: "email" } ], conditions: [ { when: "accountType", is: "business", show: true } ] } ], options: { submitLabel: "Submit", validateOnBlur: true } };
Conditional logic

Conditions are evaluated by the execution engine on every state change. They can control visibility, required status, or disabled state.

// Compound condition — AND logic conditions: [ { logic: "AND", rules: [ { when: "plan", operator: "equals", value: "enterprise" }, { when: "seats", operator: "greaterThan", value: 10 } ], effect: "show" } ]

Open design questions
Q1Hidden field value retention
Should hidden field values be retained in state but excluded from payload, or wiped on hide? Leaning toward retain — easier to restore when a condition flips back.
Q2Async validator submit behavior
Should async custom validators block submission or only warn? Decision needed before Phase 2 ends.
Q3Field groups as first-class schema concept
Deferred to V2 unless adoption research shows it is a blocking gap.

Architecture decision log
ADR-01
Zustand over Redux for state management
Alternatives considered: Redux Toolkit · Jotai · Valtio
Redux is overkill for a library. Zustand's subscribe API works outside React, which the engine needs for framework-agnostic operation. Footprint is ~1 KB vs ~10 KB for RTK.
ADR-02
Custom validation engine, not Yup or Zod as core
Alternatives considered: Yup · Zod · Valibot
Keeps the core bundle small and dependency-free. Zod is available as an opt-in adapter. Yup was rejected due to its mutation semantics conflicting with Immer.
ADR-03
tsup over raw Rollup for library bundling
Alternatives considered: Rollup · esbuild-only · webpack
tsup wraps esbuild with automatic .d.ts emission and handles ESM/CJS dual output with zero configuration. Saves several hundred lines of Rollup config.
ADR-04
TypeScript-first, no plain JS source
Alternatives considered: Flow · JSDoc on JavaScript
TypeScript types are the schema's canonical interface. Teams rely on IDE autocomplete to discover valid field properties — this only works well with TypeScript source.
ADR-05
Turborepo over Nx for monorepo orchestration
Alternatives considered: Nx · Lerna · pnpm workspaces only
Turborepo's caching model is simpler to configure for this repo's shape. Nx adds generator overhead not needed at FormForge's current scale.
